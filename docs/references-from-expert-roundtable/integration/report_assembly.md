# 报告拼装(Report Assembly)

> **说人话**:这份文件管"最后一份报告怎么拼出来"。规则一句话——圆桌报告就是全部(固定三段:一、需求背景 / 二、页面测评 / 三、圆桌纪要),**⛔ 不再有任何附录**。HCI 热力图与客观读数全部内嵌进第二节「页面测评 → 客观数据速览」的表格里;有几个界面要分析,就跑几次 phase A、把各自的热力图嵌进「单页面分析表」对应行。报告止于第三节「圆桌纪要 → 决策建议」+ 全文引用灰字。

---

## 一、最终交付形态(默认)

**一份飞书 docx**,只有三段正文,**无附录**:

```
【圆桌评审】<对象简称> - YYYY-MM-DD
├── 一、需求背景        ← 💡 橘色 callout 点明产品定位+主链路 / 业务起因 / 核心改动与最终方案
├── 二、页面测评        ← 子级① 客观数据速览(Phase A 客观:单页面分析表[页面热力图(直接嵌图)/界面客观描述(指标带解释)/对用户的影响] + 跨页面动线分析表)/ ② 单页面可理解性走查(Phase B 产品专家逐页判断,单张大表 + 每页 colspan 分组标题行)/ ③ 页面衔接可理解性(Phase B 产品专家逐对相邻页面判断,单张大表 + 每对 colspan 分组标题行,仅多界面有)
└── 三、圆桌纪要        ← 子级(顺序固定):共识要改的 / 有分歧的(各角色立场)/ 共识做的好的 / 待办总结(P0/P1/P2)/ 决策建议(phase C 张力通俗一句话 + 🅰/🅑 决策路径 + 决策偏好因子纯名称)

(报告到此结束 + 全文引用灰字。⛔ 无任何附录)
```

> **框架不可改**:正文一级标题固定三段(一、需求背景 / 二、页面测评 / 三、圆桌纪要),不得增删或重命名。**⛔ 不得添加任何附录**(不再有「附录 0 / 附录 A/B/…/N」整套机制)。

模板见 `references/phase_b_roundtable/report_template_xml.md`。

## 一·B、多界面 HCI 分析(关键)

当用户提交**多个界面**(多张截图 / 多个 URL / 一个流程的多个步骤)时:

1. **逐个界面跑 phase A**:对每个界面单独运行 `scripts/generate_heatmap.py`,各自得到独立的 `heatmap.png` + `metrics.json`。建议产出物按界面编号隔离落盘:`hci_1/`、`hci_2/`…(各含 `heatmap.png`、`metrics.json`)。
2. **热力图全部进「单页面分析表」(block_replace 进单元格)**:每个界面在第二节「客观数据速览 → 单页面分析表」里占一行,该界面的注意力热力图**直接嵌入第一列「页面热力图」单元格**(界面名 bullet 下方),落库后用 `block_replace` 把整单元格换成内含 `<img src="token" width=".." height=".."/>` 的单元格(图进单元格、保持原比例)。有几个界面就加几行,⛔ 不另起附录、⛔ 不用 selection-with-ellipsis。
3. **指标必须带解释**:第二列「界面客观描述」逐界面列 4 项 HCI 指标,每个指标**不能只写数值**,必须紧跟一句"这个数值的高/低意味着什么"(对用户体验的客观含义)。
4. **跨页面读数进「跨页面动线分析表」**:多界面时第二节追加一张 2 列表(观察 / 逐页读数),装动线接力、认知负荷走势、首屏可见度走势、注意力集中度走势(只测不判)。
5. **共享给角色的是全集**:phase B 角色发言前,把**所有界面**的 HCI 关键结论 + 各自 metrics.json + semantic.json 一并共享(见 `shared_knowledge.md`),角色判断需说明针对的是哪个界面。
6. **单界面时**:单页面分析表只一行,跨页面动线分析表可省略。**无论几个界面,都不产出附录。**

## 一·C、跨页面读数(多界面专属)

当输入 **≥2 张关联界面**时,跑 `scripts/cross_page_summary.py` 得跨页面 4 项客观度量(动线接力一致性 / 注意力衰减曲线 / 视觉风格连贯度 / 路径深度与负荷累加)+ 业务目标对齐表,指标定义与红线见 `references/phase_a_hci/cross_page_metrics.md`。它回答的是单页读数回答不了的问题:**这几页连起来,体验连不连贯、能不能达成业务目标。**

**两层分工(关键,不破红线):**
- **客观层(进第二节「跨页面动线分析表」,Phase A 只测不判)**:跨页面 4 项客观度量的逐页数值/曲线,只摆事实。
- **判断层(进第三节「圆桌纪要」,Phase B/C 只评不裁)**:"是否连贯、能否达成业务目标"是**判断**,由三角色(投手看动线引导、产品看架构断层、商家看老板能否走到底)在共识/分歧子级给出,若"连贯 vs 转化效率"成张力则在「决策建议」子级收敛。

> **顺序与业务目标若为推测必须显式标注**:页面顺序由技能推测、业务目标从 PRD 抽取或用户输入,均要在表里标注"⚠️ 顺序为技能推测"。⛔ 跨页面读数不再单独成附录,只作为第二节「跨页面动线分析表」的数据。

## 二、热力图怎么进表格(关键 · 最易出错 · 两条硬性红线)

热力图**只嵌一个地方**:第二节「客观数据速览 → 单页面分析表」第一列「页面热力图」单元格内。

> **🔴 硬性红线 1:图片必须落在 `<td>` 单元格内部**(最易出错)。⛔ 严禁让图成为表格之后的兄弟块/独立段落。
> **🔴 硬性红线 2:图片必须保持原始宽高比**。`<img>` 必须按原图真实像素宽高等比设置 width/height,⛔ 不得被单元格拉伸变形,⛔ 不得只给 width 不给 height。

**标准嵌图流程(block_replace,不要用 selection-with-ellipsis):**

1. 模板里该单元格先用文本占位符 `IMG_HEAT_S1`(界面 1)、`IMG_HEAT_S2`(界面 2)…落库。
2. 嵌图前先用 `identify`(或 PIL)读出每张热力图的**原始像素宽高**,按目标列宽(单页面分析表第一列 ≈220)等比换算出 width/height。
3. 用 `lark-doc` 的 `+media-insert --file <界面1热力图路径>`(或 `+upload-file`)把本地热力图上传飞书,**仅用于拿到 image token**。
4. 用 `+fetch --detail with-ids` 定位含占位符 `IMG_HEAT_S1` 的**单元格 block_id**。
5. 用 `block_replace` 把该整个 `<td>` 单元格 block 替换为内含 `<img src="token" width=".." height=".."/>` 的单元格 XML——图随之落进单元格内、保持原比例。逐界面替换 S1/S2/…
6. ⛔ **禁止**用 `+media-insert --selection-with-ellipsis 'IMG_HEAT_S1'` 嵌单元格图:该方式会把图作为**表格之后的独立兄弟块**插入(图跑到表格外)= "图没进表格"的根因。
7. ⛔ **热力图只嵌客观数据速览表,不贴原始设计稿截图**(原图仅供工程上生成热力图)。⛔ **不放正文别处、不另起附录**。

---

## 二·B、问题标注图怎么进表格(走查列专属 · 唯一放原图的地方)

第二节「单页面可理解性走查」表的**第 2 列「页面问题标注」**,是全文**唯一允许放红框标注原图**的位置(「图片只放热力图」铁律的唯一例外)。**同样遵守两条图片硬性红线(进单元格 + 保持原比例)。**

1. 模板里该列每行先用占位符 `IMG_ISSUE_Sx_NN`(x=界面序号,NN=走查项序号 01–05)落库。
2. 产品专家给出每条走查项**出问题的模块序号清单** → 写 `issues.json`;跑 `scripts/annotate_issues.py --image 原图 --semantic semantic.json --issues issues.json --out issue_annotated_Sx_NN.png`,在出问题模块的 **bbox**(Phase A 客观坐标)上画**模块级红框**。
3. 嵌图前先用 `identify`/PIL 读红框图原始像素宽高,按目标列宽(走查表「页面问题标注」列 ≈240)等比换算 width/height。
4. 用 `+media-insert --file <红框图路径>`(或 `+upload-file`)上传**拿 image token**;`+fetch --detail with-ids` 取含 `IMG_ISSUE_Sx_NN` 占位符的单元格 block_id;用 `block_replace` 把该整个 `<td>` 替换为内含 `<img src="token" width=".." height=".."/>` 的单元格(图进单元格、保持原比例)。⛔ **不用** `+media-insert --selection-with-ellipsis`(会插成表格后的兄弟块)。
5. **该走查项无问题** → 删除占位符 / 填「—」,不嵌图。
6. **坐标归 A、判断归 B**:bbox 是 Phase A 客观事实,"哪个模块有问题"是 Phase B 产品专家判断,画框只把 B 的判断标到 A 的坐标上,不新增判断。

## 三、主体边界(防喧宾夺主)

- 圆桌报告 = 决策结论 + 三方观点 + 待办总结 + 客观数据,是用户**要拿去决策**的部分,全部在三段正文里。
- HCI 客观读数 = 支撑证据,**内嵌**在第二节「客观数据速览」表格里(热力图 + 4 指标带解释),不喧宾夺主、不另开附录。
- 正文引用 HCI 数据时,指标值后必须带"高低意味着什么"的客观解释,不堆生数字。

## 四、拼装顺序(主持人执行)

1. phase A 跑完 → 拿到每个界面的 `heatmap.png` + `metrics.json` + `semantic.json`。**多界面则对每个界面各跑一次,按界面顺序编号(界面 1…N);并跑跨页面汇总,得到跨页面客观度量 + 业务目标对齐表。**
2. phase B 跑完 → 得到共识区/分歧区 + 单页面可理解性走查结论(逐页)+ 页面衔接可理解性走查结论(逐对相邻页面,仅多界面)。
3. phase C 跑完 → 得到决策建议(核心张力通俗一句话 + 🅰/🅑 两条决策路径 + 决策偏好因子纯名称)。
4. 按模板填三段正文(一、需求背景 / 二、页面测评 / 三、圆桌纪要):
   - 第二节「页面测评」子级①「客观数据速览」= 💡 橘色总览 callout + 单页面分析表(每界面一行,第一列嵌热力图 IMG_HEAT_Sx 占位、第二列指标带解释)+ 多界面追加跨页面动线分析表;子级②「单页面可理解性走查」由产品专家基于 semantic.json **逐页**出 5 条清单走查结论,**单张大表 + 每页一个 `<td colspan="4">` 分组标题行**(加序号 + 加粗),**第 2 列「页面问题标注」用 IMG_ISSUE_Sx_NN 占位**(有问题才嵌红框图);子级③「页面衔接可理解性」(仅多界面)由产品专家**逐对相邻页面**判断前后页预期落差,**单张大表 + 每对一个 `<td colspan="3">` 分组标题行**(格式「页面A → 页面B」,加序号 + 加粗),3 列(走查项 / 严重度 / 判断,无「页面问题标注」列)。
   - 第三节「圆桌纪要」按顺序填 共识要改的 → 有分歧的 → 共识做的好的 → 待办总结 → 决策建议。
5. 用 lark-doc `+create --api-version v2 --doc-format xml` 落库。
6. 落库后嵌图(两条硬性红线:进单元格 + 保持原比例):用 `+media-insert`/`+upload-file` 上传热力图/红框图拿 image token → `+fetch --detail with-ids` 取占位符所在单元格 block_id → `block_replace` 把整单元格替换为内含 `<img src="token" width=".." height=".."/>` 的单元格。先嵌每界面热力图(IMG_HEAT_Sx),再对**有问题的走查项**跑 `annotate_issues.py` 生成红框图并嵌进走查表第 2 列(IMG_ISSUE_Sx_NN)。⛔ 全程不用 `+media-insert --selection-with-ellipsis` 嵌单元格图。
7. 正文末尾 `<hr/>` + 一行灰字汇总全文引用切片。回传 URL。**⛔ 全程不拼任何附录。**

## 五、红线
- **⛔ 不产出任何附录**:报告只有三段正文,HCI 热力图与客观读数全部内嵌第二节「客观数据速览」表格,报告止于第三节「决策建议」+ 全文引用灰字。
- **第二节客观在前、判断在后**:子级① 客观数据速览(Phase A 客观事实,只测不判)在前,子级② 单页面可理解性走查(Phase B 产品专家逐页判断,只评不裁)、子级③ 页面衔接可理解性(Phase B 产品专家逐对相邻页面判断,仅多界面)在后,两层严格分工不可混淆。
- **能用表格就用表格(铁律)**:单页面可理解性走查 + 页面衔接可理解性都用**单张大表 + colspan 分组标题行**(分组标题加序号 + 加粗),⛔ 不退化为"多张小表 + 文字小标题"。
- **🔴 图片必须进单元格(硬性红线)**:所有图片(热力图、红框图)**必须落在目标 `<td>` 单元格内部**,落库后用 `block_replace` 把含占位符的整单元格 block 替换为内含 `<img src="token"/>` 的单元格 XML;⛔ 严禁用 `+media-insert --selection-with-ellipsis` 嵌单元格图(会插成表格后的兄弟块)。
- **🔴 图片必须保持原始宽高比(硬性红线)**:`<img>` 必须按原图真实像素宽高等比设置 width/height,⛔ 不得被单元格拉伸变形、⛔ 不得只给 width 不给 height;先用 identify/PIL 读原图像素宽高,按列宽等比换算。
- **热力图嵌入位置铁律**:每个界面的注意力热力图**只嵌「单页面分析表」第一列「页面热力图」单元格内**,⛔ 不放正文别处、⛔ 不另起附录。有几个界面就加几行。
- **指标必须带解释**:「界面客观描述」列每个 HCI 指标不能只写数值,必须紧跟一句"这个数值的高/低意味着什么"(对用户的客观含义)。
- 不得因任何原因压缩 phase C 决策建议——决策建议是圆桌的最终价值落点。
- **决策建议样式不偏离**:标题为「决策建议」(不带"双画像"字样);核心张力用通俗一句话讲清在纠结什么;决策偏好因子用纯名称、不带 A1/B3 等 ID 序号。
- **结论性/矛盾性描述要高亮**:凡结论性判断(议题判断、走查判断、共识/分歧定性、决策倾向)与具矛盾性/需关注的描述(冲突点、风险点、与 HCI 数据相悖处),用黄色文字高亮 `<span background-color="yellow">…</span>` 或加粗 `<b>…</b>` 突出。
- **全文语义通顺、无黑话**:报告禁止黑话/行话/未解释简写;「N 屏 / N 态 / N 档」等压缩表达首次出现必须展开(如"4 屏"→"4 个界面 / 4 屏页面")。
- **图片只放热力图(唯一例外:走查问题标注列)**:正文涉及界面图时**只嵌注意力热力图**,不贴原始设计稿截图;**唯一例外**是第二节「单页面可理解性走查」表第 2 列「页面问题标注」——该列允许嵌**红框标注原图**(原始截图 + 红框框出出问题的模块),用于直观指出问题位置。
- **高亮框用橘色**:需要突出的 callout(核心张力、重要提示)统一用 `light-orange/orange`,不用灰色底。
- **能用表格就用表格**:可结构化对比/罗列的内容一律用表格,不退化为长 bullet。
- **禁止元描述标签**:不要把"一句话结论/结论先行/一句话总结/一句话总览"当成标题或表头写出来(内在逻辑仍要结论放最前);✅ wiki 模板里的"结论:/观点:/建议:"内容前缀保留。
- **标题编号**:H1 用中文数字(一、二、三…),H2 用带圈数字(①②③…);正文罗列用有序列表。
- **切片 ID 视觉弱化(形式 C)**:正文不内联 `id: xxx_062_02`;每节末尾用一行灰字汇总「本节引用:xxx_062、xxx_071」。
- **跨页面读数只测不判**:跨页面动线分析表的客观度量、业务目标对齐只摆事实,"连贯/达成"的结论归第三节圆桌纪要;页面顺序、业务目标若为推测必须显式标注。
