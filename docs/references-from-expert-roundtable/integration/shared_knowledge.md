# 共享客观知识协议(Shared Knowledge)

> **说人话**:这份文件解决一件事——phase A 机器量出来的客观数据,怎么"喂"给三位专家,让他们的判断都建立在同一份事实上,而不是各说各话、各凭印象。

---

## 一、什么是"即时生成的客观知识"

phase A(HCI 分析)对**本次被测界面**现场生成的数据:
1. **热力图**(注意力分布,deep saliency 引擎产出)
2. **4 个核心指标**(见 `references/phase_a_hci/metrics_theory.md`),存于 `metrics.json`
3. **HCI 度量报告**(对上述的事实性陈述,只测不判)
4. **页面语义采集 `semantic.json`**(只测不判):对界面做 OCR + 结构标注,逐模块记录「序号 / 文案 / 类型」、各模块「承诺(claims)」、模块间「关系(relations:并列/因果/解锁依赖/指代,只标类型不下判断)」、「自上而下阅读序(reading_order)」。这是**事实采集**(看见了什么、写了什么、谁指向谁),不判断通顺/合理。定义见 `references/phase_a_hci/metrics_theory.md`「⑤ 页面语义层」。

它与角色 knowledge_base(静态、提前标注的行业经验)互补:knowledge_base 回答"行业一般怎么看",HCI 数据回答"这个界面实际怎么样"。

---

## 二、共享时机与对象

- **时机**:phase B 阶段 1「角色独立陈述」**之前**。
- **对象**:全部 3 个角色(投手 / 产品 / 本地商家),**无差别共享同一份**。
- **形式**:把 HCI 报告(或其关键结论摘要 + metrics.json 数值)+ **页面语义采集 `semantic.json`** 作为只读上下文,注入每个角色的独立陈述输入。

> **`semantic.json` 一并随附共享**:它是 Phase A 的语义层**事实**(只测不判),供产品专家在 Phase B 做「单页面可理解性走查」(逐页)与「页面衔接可理解性走查」(逐对相邻页面,仅多界面)的判断底座(单页:读不读得懂、模块关系合不合理;衔接:从前页跳后页预期是否有落差)。事实归 A、判断归 B,角色不得篡改语义层事实。

---

## 三、各角色如何使用同一份客观数据(同数据 · 不同视角)

| 角色 | 看 HCI 数据时关注 |
|---|---|
| 投手 | 核心转化入口在不在热区?信息密度会不会拖累点击决策?动线是否引导到留资/下单? |
| 产品 | 注意力分布是否符合信息架构预期?关键功能是否被淹没?层级清晰度指标是否达标? |
| 本地商家 | 门店老板一眼能不能看到该看的?入口是不是藏得太深、操作要不要培训? |

**同一份热力图,三方读出三种判断**——这正是 phase C 决策张力的客观源头。

---

## 四、注入约定(实现细节)

1. phase A 跑完后,产出物落在当前工作目录:`heatmap.png`、`metrics.json`、HCI 报告(飞书文档 URL 或本地 md)。
2. phase B 召唤角色时,在每个角色的 prompt 中附:
   - HCI 报告关键结论(3-5 条事实)
   - `metrics.json` 的 4 个指标数值
   - `semantic.json` 的页面语义采集(模块清单 / 承诺 / 关系 / 阅读序)
   - 热力图(如角色支持读图)或其文字化描述
3. 角色引用 HCI 数据时,**标注来源为 phase A**(区别于引用自身 knowledge_base 的 `id_xxx` 切片)。

---

## 四·B、多界面注入(关键)

当本次评审涉及**多个界面**时:

1. **逐界面跑 phase A**:每个界面各跑一次 `generate_heatmap.py`,产出物按界面隔离落盘(`hci_1/`、`hci_2/`…,各含 `heatmap.png` + `metrics.json`)。
2. **共享的是全集,不是其一**:phase B 角色发言前,把**全部界面**的 HCI 关键结论 + 各自 metrics.json 一并注入每个角色,**无差别共享**。不可只给某一个界面而隐藏其余。
3. **角色引用须指明界面**:角色引用 HCI 数据时,除标注来源 phase A 外,还必须标明**针对哪个界面**(如「界面 2 的首屏注意力…」),避免跨界面数据混淆。
4. **允许跨界面对比观察**:角色可基于多界面数据做横向对比(如「界面 1 入口在热区、界面 2 入口偏冷」),但每条对比都要可回溯到对应界面的 metrics.json。
5. **全部界面热力图进同一张表**:多界面时,各界面热力图分别嵌进圆桌报告第二节「客观数据速览 → 单页面分析表」对应行(每界面一行),客观读数同表呈现;⛔ 不另起任何附录(见 `report_assembly.md`)。
6. **跨页面客观度量一并共享**:除单页指标外,把**跨页面 4 项客观度量**(动线接力一致性 / 注意力衰减曲线 / 视觉风格连贯度 / 路径深度负荷)+ 推测的页面顺序 + 业务目标,作为只读上下文注入每个角色(定义见 `references/phase_a_hci/cross_page_metrics.md`)。
7. **连贯性与业务达成由角色判断**:跨页面度量是**事实**,"体验连不连贯、能否达成业务目标"是**判断**——必须由三角色给出(投手看动线引导是否到留资/下单、产品看跨页架构有无断层、商家看老板能否一路走到底),Phase A 不下此结论。角色判断须挂靠具体跨页面数值(如「衰减曲线在第 3 步骤降 0.3 → 我认为此处可能断流」)。

---

## 五、红线
- HCI 数据是**事实底座**,角色不得篡改或选择性无视;若某角色判断与客观数据冲突,必须显式说明理由。
- 角色**不得**把"主观判断"伪装成"客观数据";凡引用 phase A 数值必须可回溯到 metrics.json。
