# 跨页面客观度量(Cross-Page Metrics · 只测不判)

> **说人话**:一张图只能看"这一页用户看哪";但一个流程是好几页串起来的。这份文件定义"把好几页连起来看"时,机器要量的几个客观数字——比如用户的注意力是一路被引导下去还是中途断了、两页之间的入口对不对得上、几页风格统不统一。**注意:这里只量数字、不下"好不好"的结论**;连不连贯、能不能达成业务目标,交给 Phase B 三角色 + Phase C 主持人去判断。

---

## 一、什么时候算跨页面度量

- **触发**:本次评审输入 **≥2 张关联界面**(多图)时**自动**触发,无需用户显式声明。
- **前提**:先确定页面**顺序**(见下节"二·顺序推测")。顺序错,动线分析全错。
- **产出**:跨页面 4 项客观度量 + 业务目标对齐表(只测不判),作为**第二节「客观数据速览 → 跨页面动线分析表」的数据**填入圆桌报告;⛔ 不再单独成附录。

---

## 二、顺序推测(技能自己推,不问用户)

按以下优先级自动判定页面先后,**并在总览报告里显式写出"推测依据"**,供人复核:

1. **文件名/序号线索**:`step1/step2`、`1_/2_`、`首页→详情→下单` 等命名。
2. **PRD/流程描述线索**:若有 PRD,按其描述的操作流程排序。
3. **页面语义线索**:列表页→详情页→表单页→结果页 这类典型转化漏斗的语义递进。
4. **CTA 承接线索**:A 页主 CTA 文案("去下单")与 B 页标题/功能("下单页")能对上,则 A→B。

> **红线**:顺序是"推测",必须在报告里标注「⚠️ 页面顺序为技能推测,如有误请指正」,不得伪装成用户确认的事实。

---

## 三、四项跨页面客观指标(只测不判)

| 指标 | 测什么(纯客观) | 计算口径 | 数据来源 |
|---|---|---|---|
| **① 动线接力一致性** | 上一页"主热区/主 CTA 落点"坐标,与下一页"承接入口"位置的对齐程度 | 两点像素距离 + 归一化(按对角线);区域重合度 IoU | 各页 heatmap 峰值坐标 + AOI |
| **② 注意力衰减曲线** | 转化路径上,每一步首屏"核心入口可见性"指标连成的折线 | 逐页取 metrics.json 的 `entry_visibility`,按顺序成序列 | 各页 metrics.json |
| **③ 视觉风格连贯度** | 跨页"信息密度 / 层级清晰度 / 认知负荷"的波动 | 各指标跨页方差 / 极差(波动越大越不连贯) | 各页 metrics.json |
| **④ 路径深度与负荷累加** | 完成目标的步数 + 每步认知负荷的累加值 | 步数 = 页数;累加 = Σ 各页 `cognitive_load` | 流程定义 + 各页 metrics.json |

> 以上一律是**客观陈述**:只写「第 2 步可见性 0.42,较第 1 步 0.72 下降 0.30」「页 1 与页 2 的主 CTA 落点相差 380px(对角线占比 21%)」,**绝不写**「这样体验不好/会流失」——判断留给角色。

---

## 四、业务目标对齐(目标从哪来)

跨页面要回答"能否达成业务目标",**业务目标的来源**:

1. **优先从 PRD 抽取**:若有 PRD,抽取其中的目标陈述(如"提升留资转化率""降低下单流失")。
2. **退而由用户输入**:无 PRD 或抽不到时,在 Phase B 阶段 0 澄清里追问一句"本流程的核心业务目标是?"。
3. **兜底**:两者都缺时,按页面语义推测一个候选目标,并标注「⚠️ 业务目标为推测」。

产出一张**业务目标对齐表**(只摆事实,不下结论):

| 业务目标 | 路径关键步 | 该步客观指标 | (留白:角色判断) |
|---|---|---|---|
| 例:提升留资率 | 步骤3 表单页 | 表单入口可见性 0.38、认知负荷 偏高 | ←Phase B 角色在圆桌里填判断 |

> 表的最后一列**在 Phase A 留空**(只测不判),由 Phase B 三角色 / Phase C 主持人在圆桌正文里填"连不连贯、达不达成"的判断。

---

## 五、红线(继承 Phase A "只测不判")

- 跨页面度量**只给数字和事实**,不写"好/坏/应该/会导致";连贯性与业务达成的**判断归 Phase B/C**。
- 页面顺序、业务目标若为推测,**必须显式标注推测**,不得伪装成既定事实。
- 跨页面总览**不替代**单页报告;两者都要在(先总后分)。
