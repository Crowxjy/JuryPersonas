# 飞书文档输出格式规范(已与用户确认,务必遵循)

本 skill 默认产出**飞书文档**(用 `lark-doc` skill 创建)。文档格式经用户多轮确认,以下为**强制规范**,再次输出时严格按此渲染。数值来自 `metrics.json`。

## 零、内容铁律

1. 测量与数据部分**只陈述事实**,不写"好/坏/应该"。
2. **全文禁止出现"说人话"字眼**(以及任何同义白话标注词)。白话解释可保留,但不要加"说人话"这种标签。
3. 末尾「体验初判」**只指出问题在哪**,不给改版方案——多角色评审与具体改版留给专家圆桌 skill。
4. **页面语义层只测不判**:对界面做 OCR + 结构标注,只记录"看见了什么、写了什么、谁指向谁、阅读序",**不判断**"通不通顺、合不合理"——通顺与否的判断归专家圆桌(产品专家)。语义采集结果落 `semantic.json`,并在报告中以「页面语义层」模块如实呈现。
5. **语义通顺、没有黑话(全文铁律)**:报告通俗可读,**禁止黑话/行话/未解释简写**。凡"N 屏 / N 态 / N 档"等压缩表达,首次出现必须展开(如"4 屏"→"4 个界面 / 4 屏页面";"6 态权益折叠为 3 态"→"权益从 6 种状态精简为 3 种状态");专名/缩写首次给全称。每条要点**先给判断、再给依据**(把判断放最前)。
6. **只嵌热力图、不贴原始设计稿图(全文铁律)**:文档**只嵌入注意力热力图**,⛔ 不再贴原始界面截图。原始界面仅用于工程上生成热力图,不进文档正文。
7. **禁止"结论先行 / 一句话总结 / 结论 / 一句话总览"等字面标签(全文铁律)**:不要把这些词**当成标题或表头写出来**——直接把结论本身放最前就是结论先行,不需要再贴标签。
8. **能用表格就用表格(全文铁律)**:凡可结构化对比/罗列的内容(指标、清单、对照)一律优先 `<table>` 呈现,不退化为长 bullet;纯顺序步骤用有序列表。

---

## 一、文档命名

- 标题格式固定为:`HCI度量报告_<动态主题名>`
- `<动态主题名>` 根据被测界面动态生成,如「经营通界面注意力分析」。

## 二、标题层级编号

- **一级标题(H1)用中文数字:** 一、二、三、四、五、六、七…
- **二级标题(H2)用带圈数字:** ①、②、③…
- 涉及数字罗列的内容,用**有序列表**(`<ol><li seq="auto">`)表达,**不要**用"一、二、…"中文序号去做正文罗列。

## 三、一级模块顺序(H1)

固定八个一级模块,顺序如下:

1. **一、整体速览**
2. **二、注意力分布**
3. **三、视觉动线(扫描路径)**
4. **四、认知负荷(视觉复杂度)**
5. **五、首屏覆盖与注意力集中度**
6. **六、页面语义层(只测不判)**
7. **七、体验初判**
8. **八、附录说明**

> ⚠️ **不放「结构化数据 / metrics.json / semantic.json」原始文件模块**。metrics.json / semantic.json 仅用于工程上衔接圆桌 skill,**不在文档正文里贴原始 JSON**;「页面语义层」模块是对 semantic.json 的人类可读呈现(只陈述事实)。

## 四、各模块渲染规则

### 一、整体速览 —— 左图右文分栏(只放热力图)

用 **2 列分栏(`<grid>`)**:左栏放热力图,右栏放文字描述。

- **左栏(width-ratio≈0.42)**:一行说明文字(`注意力热力图(红=高 → 黄=中 → 绿=低 → 透明=冷区)`)+ **一张热力图**(⛔ 不放原始界面截图)。
- **右栏(width-ratio≈0.58)**:标题「关键现象」+ 有序列表,逐点概括全页最关键的客观现象(用户主要看哪、忽略什么、首屏承接、焦点竞争、关键入口注意力、整体认知负荷等)。把最关键的判断放最前(不要写"结论先行/一句话总结"字样)。

> 🔧 **热力图嵌入分栏的技术要点(关键,否则图会丢失)**:
> `<img href>`(网络图)直接写进 `<grid>`/`<table>` 单元格,以及 `media-insert`,都会让图片落到容器外或被丢弃。**必须用底层三步法把本地热力图塞进左栏内部**:
> 1. 在左栏(grid_column,block_type=25)下创建空图块:
>    `POST /open-apis/docx/v1/documents/{doc}/blocks/{column_id}/children`,body `{"children":[{"block_type":27,"image":{"token":""}}],"index":N}`,拿到 `block_id`。
> 2. 上传本地热力图并绑到该块:
>    `POST /open-apis/drive/v1/medias/upload_all`,`--file file=heatmap.jpg`,data `{"file_name":"heatmap.jpg","parent_type":"docx_image","parent_node":"<block_id>","size":<bytes>}`,拿到 `file_token`。
> 3. 绑定:`PATCH /open-apis/docx/v1/documents/{doc}/blocks/{block_id}`,data `{"replace_image":{"token":"<file_token>"}}`。
> 创建分栏骨架时,可在左栏放一个占位段落(如 `IMG_ANCHOR_LEFT`)定位 index,绑定完热力图后再 `batch_delete` 删除占位段。
> **自检:** 完成后用 `GET .../blocks` 统计 `block_type==27` 的数量应**恰好=1**(只一张热力图,不能重复嵌入,也不放原图)。

### 二、注意力分布

- 先一句:`本页 Top5 注意力热点:`
- 紧跟一张**表格**:列 = 排名 / 注意力占比 / 峰值 / 位置(x,y) / 对应元素(语义),按 `metrics.json` hotspots 填。
- 模块末尾,用**引用块(`<blockquote>`)** 放解释说明文案(是什么 / 理论来源 / 怎么算)。

### 三、视觉动线(扫描路径)

- 先一句:`本页预测注视顺序:`
- 紧跟**有序列表**:逐点列出热点 1→热点 5 的顺序与坐标、对应元素。
- 一段正文:概括扫视路径(自上而下跳跃、跳过了哪个关键入口)。
- 模块末尾,用**引用块**放解释说明文案(理论来源 / 算法 / 注明是热点级顺序近似)。

### 四、认知负荷(视觉复杂度)

- 先一句:`本页测量值:`
- 紧跟**有序列表**:composite(综合分)/ edge_density / color_variety / luminance_entropy / focal_competition,各一项。
- 一段正文:客观解释关键子项(如 focal_competition=2 表示有 2 个抢眼程度相当的焦点)。
- 模块末尾,用**引用块**放解释说明文案(Feature Congestion 等理论)。

### 五、首屏覆盖与注意力集中度

- 先一句:`本页测量值:`
- 紧跟**有序列表**:首屏注意力占比 / 集中度 Gini / 首屏内热点 / 热区像素比(折叠线 px),各一项。
- 一段正文:客观解释(首屏含几个热点、排名 1 是否在首屏等)。
- 模块末尾,用**引用块**放解释说明文案(NN/g 折叠线/集中度理论)。

> 📌 **引用块统一规则**:「注意力分布、视觉动线、认知负荷、首屏覆盖与注意力集中度」四个模块的"是什么/理论来源/怎么算"解释性文案,**一律放在该模块末尾**,用引用块(blockquote)呈现,正文先给数据与结论。

### 六、页面语义层(只测不判)

> 这是为「结合业务理解页面」补的客观底稿:**只采集事实,不下判断**。判断(读不读得懂、模块关系合不合理)归专家圆桌的产品专家。

- 先一个 `<callout emoji="🔎">` 提示:本模块仅客观采集页面文案与结构(看见了什么、写了什么、谁指向谁),不判断通顺/合理,通顺与否的判断归专家圆桌。
- 分四个二级标题:
  - **① 模块清单** —— 表格(序号 / 模块文案摘要 / 类型),按自上而下顺序列出页面各模块。类型如:标题 / 入口按钮 / 任务卡 / 权益卡 / 说明文案 / 状态标签 等。
  - **② 模块承诺** —— 有序列表,逐模块记录它向用户**承诺了什么**(如"完成任务可解锁权益"),只摘原文语义,不评价。
  - **③ 模块关系** —— 表格(关系 / 涉及模块 / 类型):只标关系**类型**(并列 / 因果 / 解锁依赖 / 指代),**不判断**该关系是否表达清楚。
  - **④ 自上而下阅读序** —— 有序列表,按版面从上到下列出阅读顺序(模块序号 + 文案摘要),供下游判断"顺读一遍能否理解"。
- 全模块**禁止**出现"好/坏/应该/不清晰/容易误解"等判断词;只写事实。

### 七、体验初判

- 先一个 `<callout emoji="⚠️">` 提示:本段为通用体验初判,只指出问题在哪,不是改版方案;多角色评审与改版交由专家圆桌 skill。
- 分两个二级标题:
  - **① HCI 问题** —— 有序列表,纯 HCI 角度的客观问题(首屏承接、焦点竞争、关键入口不在高注意力区等),每条**加粗结论 + 依据(引用指标数值)**。
  - **② 业务语境下的问题** —— 有序列表,结合业务/文案语义/动线的卡点(承诺 vs 门槛、模块关联弱、动线叙事断裂、零状态负向暗示等),每条**加粗结论 + 依据**。

### 八、附录说明

- 把「测量方法」「指标间关联性」作为附录,放在**最后**。
- 两个二级标题:
  - **① 测量方法** —— 正文 + 有序列表(三项还原手段:中心偏好/适度模糊、中央凹高斯平滑、UI 场景调参),末尾用**引用块**放引擎/校准/可信度标签("预测式结果,适用于上线前自查,不能完全替代真实用户眼动测试")。
  - **② 指标间关联性** —— 正文 + 有序列表(视觉特征→注意力分布→视觉动线→认知负荷→首屏覆盖的因果链),末尾用**引用块**放"③是①②的成因约束,④是结果"的总结。

## 五、模块间分隔

主要一级模块之间用 `<hr/>` 分隔(总览后、首屏覆盖后、页面语义层后、体验初判后)。

## 六、不要出现的内容

- ❌ 开头的总结/引擎/定位 callout(直接从「一、整体速览」开始)
- ❌「结构化数据 / metrics.json / semantic.json」正文模块(semantic.json 仅作工程衔接,「页面语义层」模块是其人类可读呈现)
- ❌ 任何「说人话」字眼
- ❌ 任何「结论先行 / 一句话总结 / 结论 / 一句话总览」等字面标签(直接把结论写出来即可)
- ❌ 原始界面截图(只嵌热力图)
- ❌ 任何黑话/行话/未解释简写(「N 屏 / N 态 / N 档」等首次出现必须展开)
- ❌ 「页面语义层」里出现判断词(好/坏/应该/不清晰/容易误解)——只测不判
- ❌ 多角色评审 / 具体改版方案(归专家圆桌 skill)
