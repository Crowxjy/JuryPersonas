# 圆桌工作流详细版(Phase B · 三角色)

> **说人话**:这是圆桌怎么开的流程。先把题目问清楚 → 三位专家各自背靠 HCI 客观数据独立打分(不许偷看对方)→ 主持人比对出"都说好的 / 都说要改的 / 吵起来的" → 吵不拢的最多留 3 个 → 出报告。

## 前置:三角色都先吃同一份"客观知识"
Phase A(HCI 分析)产出的热力图 + 4 指标 + HCI 报告 + 页面语义采集 `semantic.json`,是本项目的**即时生成客观知识**,在角色独立陈述前**共享给全部 3 个角色**。角色的主观判断必须建立在这份客观数据之上(见 `references/integration/shared_knowledge.md`)。`semantic.json` 是页面语义层**事实**(只测不判),供产品专家做单页面可理解性走查(逐页)与页面衔接可理解性走查(逐对相邻页面)的判断底座。

## 阶段 0:议题澄清(必经)
**Workshop 协议:三问 + 编号选项**。如用户提示词不明确,主持人必须先输出:
```
我需要先和您确认 3 件事(请用编号回复):

1. 评审对象类型?
   ① PRD 文档  ② 设计稿截图  ③ 线上产品 URL  ④ 上线界面回看  ⑤ 其他
2. 评审范围?
   ① 整体合理性  ② 某一交互流程  ③ 某一具体功能  ④ 视觉设计  ⑤ 其他
3. 您最关心什么?
   ① 转化效果  ② 用户体验  ③ 商家经营/上手成本  ④ 合规风险  ⑤ 不限,综合评估
```
3 个问题全部得到回答(或明确"不限")后,才能进入阶段 1。

## 阶段 1:角色独立陈述
主持人同时调用 **3 个角色**,**严禁**让任一角色看到其他角色的输出。每个角色先读 phase A 的 HCI 报告 + 页面语义采集 `semantic.json` + 自己的 knowledge_base,再输出 JSON:
```json
{
  "role": "ad_buyer | product_expert | local_business",
  "reasonable": [
    {"point": "...", "severity": "high|med|low", "evidence": ["id_xxx"]}
  ],
  "unreasonable": [
    {"point": "...", "severity": "high|med|low", "evidence": ["id_xxx"], "is_red_line": false}
  ]
}
```
每个角色:**3 reasonable + 3 unreasonable**(可少不可多);超过 3 个由角色自评严重度,主持人只取 top 3。

## 阶段 1b:单页面可理解性走查(产品专家主导 · 逐页评测)
与阶段 1 并行,**产品专家**额外基于 phase A 的 `semantic.json`(页面语义层事实:模块清单 / 承诺 / 关系 / 阅读序)做一份「单页面可理解性走查」,**对每个页面逐页**各跑一组 5 条清单,产出**第二节「页面测评」子级②** 的内容。这是 Phase B 的**判断**层(读不读得懂、模块关系合不合理),与 Phase A 的语义层**事实**(只测不判)严格分工——**事实归 A、判断归 B**,产品专家不得篡改语义层事实。**报告里用单张大表 + 每页一个 `<td colspan="4">` 分组标题行(加序号 + 加粗)** 承载逐页走查,⛔ 不退化为多张小表。

每个页面走查输出 **5 条清单**,每条**结论先行 + 依据(引用 semantic.json 模块/关系)+ 严重度(high/med/low)**:
1. **模块关系自洽** —— 模块间的并列/因果/解锁依赖/指代关系是否说得通、有无自相矛盾。
2. **承诺-路径衔接** —— 每个模块的承诺(claims)能否在页面内找到对应的操作路径/兑现入口。
3. **指代链完整** —— 指代关系(「该」「上述」「此」等)是否都能回指到明确对象,有无悬空指代。
4. **自上而下可读懂业务** —— 按阅读序(reading_order)从上到下读一遍,新用户能否理解这个页面在做什么。
5. **零状态/负向暗示** —— 是否存在空状态、未解释的负向措辞或易误解表达。

> 该走查不裁决、不出改版方案,只给"读不读得懂、合不合理"的判断 + 严重度;改版方向交由共识/分歧与决策建议环节。

### 阶段 1b 附加产出:页面问题标注图(红框,模块级)
走查表第 2 列「页面问题标注」要嵌**红框标注原图**。产品专家给出每条走查项**出问题的模块序号清单**(只标"哪个模块有问题",对齐 5 条走查判断),写成 `issues.json`(`{"problems":[{"block_id":2,"label":"①"}]}`);随后跑 `scripts/annotate_issues.py --image 原图 --semantic semantic.json --issues issues.json --out issue_annotated_Sx_NN.png`,用 PIL 在 semantic.json 对应模块的 **bbox**(Phase A 客观坐标)上画红框(模块级)。**坐标归 A、判断归 B**:画框只是把产品专家的问题判断标注到 A 的坐标上,不新增判断,也不改语义层事实。该走查项若无问题,本列留「—」、不嵌图。

## 阶段 1c:页面衔接可理解性走查(产品专家主导 · 逐对相邻页面 · 仅多界面)
当评审涉及 **≥2 个关联页面**时,产品专家额外做一份「页面衔接可理解性走查」,**对每一对相邻页面**(页面1→页面2、页面2→页面3…)判断**从前页跳到后页预期是否有落差**——用户在前页形成的预期(想看到什么、下一步该去哪),后页能否承接;有无断层、跳转突兀、看不到想看的内容。产出**第二节「页面测评」子级③** 的内容。**报告里用单张大表 + 每对一个 `<td colspan="3">` 分组标题行(格式「页面A → 页面B」,加序号 + 加粗)**,组内 3 列(走查项 / 严重度 / 判断),**无「页面问题标注」列**。每条**结论先行 + 严重度**,可发散更多衔接维度(预期落差 / 跳转突兀 / 信息连续性 / 操作连贯性)。**单界面时本走查省略。**

## 阶段 2:共识 / 分歧识别(三方版)
主持人比对 3 份 JSON:
```
共识做的好的 = 三方(或多数)reasonable 的交集(语义相似度 > 0.7)
共识要改的   = 三方(或多数)unreasonable 的交集
有分歧的     = 任意两方在同一议题上 reasonable 与 unreasonable 对立
```
- 三方判断越分散,越说明该议题存在决策张力 → 进 phase C。
- 分歧点超过 3 个:按 `severity_score = (high=3, med=2, low=1)` × 角色权重排序,取 top 3。
- **配套条件型议题归共识**:若各方结论一致,只是某方追加"前置条件且可同步满足",记为共识,不进分歧。

## 阶段 3:分歧深议(可选)
对每个保留的分歧:
1. 让分歧各方各用 80 字以内总结立场
2. 第 1 轮即收敛 → 移入共识区
3. 2 轮内仍不收敛 → 标记 `升级为人类决策`,主持人不裁决

## 收敛机制(硬门槛)
| 机制 | 触发条件 | 动作 |
|---|---|---|
| 分歧上限 | 识别出 > 3 个分歧 | 按严重度 × 角色权重排序,保留 top 3 |
| 投票收尾 | 任一分歧深议 ≥ 2 轮仍无共识 | 各自给最终立场 + 推荐方案,主持人不裁决 |
| 深议升级 | 多方在同一议题明确反对 ≥ 2 轮 | 触发"升级为人类决策"标签 → 交 phase C |
| 误判回收 | 某"分歧"经追问发现属配套条件 | 立刻迁移至"共识做的好的" |

## 阶段 4:报告生成
用 `report_template_xml.md` 渲染最终飞书 docx(正文第二节为「页面测评」,子级① 客观数据速览 / ② 单页面可理解性走查[逐页,colspan 分组单表] / ③ 页面衔接可理解性[逐对相邻页面,colspan 分组单表,仅多界面])。分歧部分移交 **phase C 决策透镜**,产出「决策建议」(核心张力通俗一句话 + 🅰/🅑 决策路径 + 决策偏好因子纯名称 + 倾向提示)。phase A 的热力图与客观读数全部内嵌进第二节「客观数据速览」表格,⛔ 不再有任何附录(见 `references/integration/report_assembly.md`)。**图片两条硬性红线:必须进 `<td>` 单元格内(block_replace 嵌图,⛔ 不用 selection-with-ellipsis)+ 必须保持原始宽高比(`<img>` 带等比 width/height)。**
