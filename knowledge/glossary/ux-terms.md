# UX 设计术语表

> 该文件被 `personas/ux-designer-*.md` 的 `imports` 字段引用,作为 UX 设计师角色公共术语基础。
> 编排原则:**先记词,再记定义,最后记反例**。每一条都附"扮演角色时怎么用",防止生搬硬套。

---

## 一、流程与方法论缩写

| 术语 | 全称 / 含义 | 扮演角色时怎么用 |
|---|---|---|
| **HMW** | How Might We,定义问题的提问框架 | 收敛阶段口头禅:"我们怎样才能让 XXX 在 YYY 场景下做到 ZZZ" |
| **KJ 法 / 亲和图** | 把零散信息归类成主题 | 评审时提议"先把这堆痛点亲和图一下" |
| **Kano 模型** | 把需求分基础型 / 期望型 / 兴奋型 | 砍需求时用 |
| **Journey Map** | 用户旅程地图 | 讨论端到端体验时常用 |
| **HTA** | Hierarchical Task Analysis,层次任务分析 | 复杂工具型产品方案设计时用 |
| **Lo-fi / Hi-fi** | 低保真 / 高保真原型 | 评审阶段对应使用 |
| **AB Test** | 灰度对比实验 | 上线验证阶段标配 |
| **VOC** | Voice of Customer,体验管理平台收集用户声音 | 上线后复盘 |
| **ROI** | Return on Investment,投入产出比 | 评估方案/砍需求时反复用 |
| **MVP** | Minimum Viable Product,最小可行产品 | 0→1 项目早期 |

## 二、可用性与体验经典原则

| 术语 | 含义 | 关联人物 |
|---|---|---|
| **启发式评估**(Heuristic Evaluation) | 用专家方式按 10 条原则做体检 | 尼尔森 |
| **Nielsen 十原则** | 见知识库《UX 设计流程》§4.1 | 尼尔森 |
| **行动七阶段** | 目标→计划→确认→执行→感知→解释→比较 | 诺曼 |
| **执行鸿沟 / 评估鸿沟** | 用户意图 vs 系统能力 / 系统状态 vs 用户理解 | 诺曼 |
| **Affordance**(示能性) | 物件本身暗示如何使用 | 诺曼 |
| **Signifier**(意符) | 显式提示在哪、怎么用 | 诺曼 |
| **Feedback**(反馈) | 操作后的可感知响应 | 诺曼 |
| **Mapping**(映射) | 控件与结果的对应关系 | 诺曼 |
| **三层情感**(本能/行为/反思) | 设计的三个情绪触达层 | 诺曼 |

## 三、用户研究与建模

| 术语 | 含义 |
|---|---|
| **Persona / 用户画像** | 一个具体虚拟人代表一类真实用户(库珀提出) |
| **Scenario / 场景** | Persona × Goal × Context 的故事化推演 |
| **同理心地图** | 列出用户「想 / 说 / 做 / 感」 |
| **认知走查**(Cognitive Walkthrough) | 模拟新手心智走流程找断点 |
| **可用性测试**(Usability Testing) | 找真实用户做任务、看出错点 |
| **桌面研究**(Desk Research) | 资料汇总,无须接触用户 |

## 四、决策与认知模型

| 术语 | 含义 / 在 UX 中的作用 |
|---|---|
| **Bounded Rationality**(有限理性) | 西蒙提出。承认用户决策有认知带宽限制,UX 不能假设"理性人" |
| **Satisficing**(满意即可) | 西蒙提出。用户不追最优,只要"够好",所以 UX 要把"够好"放到显眼位置 |
| **Design Thinking**(设计思维) | 西蒙奠基,IDEO/d.school 推广。Empathize/Define/Ideate/Prototype/Test |
| **目标导向设计**(Goal-Directed Design) | 库珀提出。锁定用户目标,而不是任务清单 |

## 五、衡量与验证

| 术语 | 含义 |
|---|---|
| **北极星指标** | 长期最重要的单一指标,所有动作向它看齐 |
| **转化漏斗** | 多步流程的逐级流失分析 |
| **埋点** | 在交互节点埋数据采集点 |
| **量表**(SUS / NPS) | 用户满意度的标准化打分 |
| **设计目标** | 由用户/业务/产品目标推导,UX 自己背的可衡量目标 |

## 六、设计原则的"五要素"

任何 UX 评审都绕不开:**易学习 / 易使用 / 易识别 / 复用性 / 一致性**。

## 七、典型反模式（资深 UX 一眼能看出的"坏味道"）

- "我觉得这样好看" — 缺依据
- "用户应该会这么用" — 缺验证
- "竞品也是这么做的" — 缺判断
- "等上线看数据吧" — 缺前验
- "这个细节 PM 不让动" — 没分清谁在背设计目标
- "这套方案能解决所有用户" — 没做用户分层

## 八、易混淆术语对照

| 容易混的概念 | 区别 |
|---|---|
| 用户目标 vs 业务目标 | 前者是用户想达到的状态;后者是公司想拿到的指标 |
| 产品目标 vs 设计目标 | 前者是产品 PM 主导的整体目标;后者是 UX 派生出的可衡量目标,服务前者 |
| Persona vs 用户分层 | 前者是「一个具体虚拟人」;后者是「一组划分维度」 |
| 痛点 vs 机会点 | 痛点 = 用户当下受阻;机会点 = 产品可以差异化的位置 |
| 启发式评估 vs 可用性测试 | 前者是专家走查;后者是真实用户做任务 |
