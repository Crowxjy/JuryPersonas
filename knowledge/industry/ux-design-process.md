# UX 设计流程共享知识库

> 该文件被 `personas/ux-designer-*.md` 的 `imports` 字段引用,作为所有 UX 设计师类角色的公共知识基础。
> 凝练自团队内部沉淀（设计思考阶段图）+ 行业经典理论（尼尔森、诺曼、库珀、西蒙）,聚焦"产品设计每个阶段 UX 都在思考什么"这条主线。

---

## 一、设计的本质定义

**设计 = 为了达成某个目标 / 解决某个问题,而进行的一系列思考并通过不同方式将之实现的过程。**

两条核心链路（可同时成立,看入口）:

- **问题视角**:发现问题 → 定义问题 → 解决问题
- **目标视角**:挖掘目标 → 拆解目标 → 达成目标

四种目标必须分清,绝不混用:**用户目标 / 业务目标 / 产品目标 / 设计目标**。设计目标是前三者的派生物,不是凭空生成。

---

## 二、UX 全流程的六个阶段

每个阶段对应"为什么做(Why)/思考什么(What)/用什么方法(How)"的三段拆解。

### 阶段 1：研究 / 发现问题

| 子目标 | 核心思考问题 | 主要方法 |
|---|---|---|
| 了解业务背景,确立业务目标 | 需求产生背景?市场空间?竞争壁垒?商业模式?北极星指标? | 利益相关者访谈、桌面研究、数据分析 |
| 了解用户,构建同理心 | 面向哪些用户?核心诉求?深层目标?当下是否成功达成?如何达成的? | 用户调研、场景分析、用户画像、同理心地图 |
| 了解产品与竞品,发现痛点机会点 | 产品现状?竞品做法、问题、借鉴点?痛点 vs 机会点? | 认知走查、Journey Map、竞品分析、数据分析 |

### 阶段 2：定义

聚焦本次迭代,把模糊的"问题/机会"收敛成可执行的"设计目标"。

- 思考:问题重要性?优势/困难/资源?设计目标是什么?要解决哪些问题?迭代范围与优先级?
- 方法:**HMW**(How Might We)定义问题、KJ 法 / 亲和图、Kano 模型、和产品经理共同评估 ROI

### 阶段 3：灵感探索

- 思考:前期信息中可复用什么?解决问题的关键点?用户心智?有无可参考的研究/模型?
- 方法:设计背景与思路评审、场景分析、行为分析、转化漏斗分析、层次任务分析(HTA)、架构设计、头脑风暴、流程设计、草图设计

### 阶段 4：方案设计

| 子目标 | 关键思考 | 方法 |
|---|---|---|
| 产出设计 demo | 是否符合产品的设计原则与规范(易学习、易使用、易识别、复用性、一致性)?技术可行性与成本预估? | 线框图、Lo-fi 原型 |
| 验证(前验) | 方案是否真有效、有依据?对应哪些任务?是否贴合设计目标?评审中收到的问题严重程度/频率/优先级?是否需要 User Testing? | 设计评审、启发式评估、可用性测试 |

### 阶段 5：方案落地

- 思考:目标和策略是否准确传达给合作方?埋点 / 实验是否到位?边界 case(在 X 场景下用户**会**做什么、**能**做什么)?可行性受阻时的关键点和替代方案?
- 方法:项目和时间管理、交互文档、交互自查表
- 角色:做好产品 / UI / 视觉 / 研发 / 测试 之间的纽带桥梁

### 阶段 6：上线验证期

- 思考:产品目标完成情况?用户反馈?表现无显著变化时是否扩大实验周期?表现不佳时是完全有问题还是部分可保留?
- 方法:灰度 / AB Test、量表、数据分析、VOC 体验管理平台、复盘会议

## 三、UX 设计师的能力三件套

| 维度 | 包含 |
|---|---|
| **Thinking 思考方式** | 目标导向、场景分析、结果验证、ROI 意识 |
| **Ability 执行能力** | 版式 / 视觉 / 动效 / 软件 / 用研 / 数据分析 |
| **Knowledge 底层知识** | 心理学(行为模型)、统计学、美学、动效知识 / 基础设计规范 |

> **重要提醒**:原则与方法论是经验沉淀的复用模型,**不是通用定理**。脱离本源内容机械套用,容易"学了不会用、不分情况滥用"。

## 四、行业经典理论锚点（蒸馏版）

UX 设计师在思考时,常显式或隐式调用以下四组理论。

### 4.1 雅各布·尼尔森（Jakob Nielsen）—— 启发式评估十大可用性原则

可用性的"通用体检清单",用于设计评审与启发式评估:

1. 系统状态可见性(System status visibility)
2. 系统与现实世界匹配(Match between system & real world)
3. 用户控制与自由(User control & freedom)
4. 一致性与标准化(Consistency & standards)
5. 错误预防(Error prevention)
6. 识别优于回忆(Recognition rather than recall)
7. 灵活性与高效性(Flexibility & efficiency)
8. 美观与简约设计(Aesthetic & minimalist design)
9. 帮助用户识别、诊断与恢复错误(Help users with errors)
10. 帮助文档与说明(Help & documentation)

### 4.2 唐·诺曼（Don Norman）—— 以用户为中心的体验设计

- **行动七阶段**(Seven Stages of Action):目标 → 计划 → 确认 → 执行 → 感知 → 解释 → 比较。两个"鸿沟"是 UX 痛点的根源:
  - **执行鸿沟**(Gulf of Execution):用户想做的 vs 系统允许的
  - **评估鸿沟**(Gulf of Evaluation):系统状态 vs 用户能否理解
- **示能性 / 意符 / 反馈 / 映射 / 约束**(Affordance / Signifier / Feedback / Mapping / Constraint):五大设计抓手
- **三层情感**(Visceral / Behavioral / Reflective):本能层、行为层、反思层,UX 不止"好用",还要"好感"和"难忘"

### 4.3 阿兰·库珀（Alan Cooper）—— 目标导向设计

- **目标 ≠ 任务**:任务会变,目标稳定。设计应锚定"用户最终想达到什么状态",而不是当下完成哪几步操作
- **Persona(用户画像)**:用一个具体的虚拟人代表一类真实用户,避免"想象的全体用户"陷阱
- **Scenario(场景)**:Persona × Goal × Context,用故事而非功能列表来推演方案
- 经典反模式:别让设计师/工程师当默认用户(*The Inmates Are Running the Asylum*)

### 4.4 赫伯特·西蒙（Herbert A. Simon）—— 设计的科学基础

- **Bounded Rationality(有限理性)**:用户决策受认知带宽、信息可得性、时间约束限制,不是经济学假设的"完美理性人"
- **Satisficing(满意即可)**:用户不寻求最优解,只要"够好"即停止。UX 必须把"够好"放在显眼位置,把"最优"留给愿意深挖的人
- **Design Thinking 原型**:西蒙将"设计"上升为"如何把现状变成期望状态"的通用学科,奠定了今天 IDEO 等机构所推广的设计思维流程
- **The Sciences of the Artificial**:复杂系统由简单规则的层级组合产生,UX 设计本质是构建一个"可被人理解的人造环境"

## 五、设计思维流程（Design Thinking 通用骨架）

斯坦福 d.school / IDEO 推广的五段式,本质是上面六阶段的高度浓缩,常用于跨职能工作坊:

`Empathize 共情 → Define 定义 → Ideate 创意 → Prototype 原型 → Test 测试`

非线性,允许任意阶段回退。

---

## 六、情景分析（Scenario Analysis）

把用户、目标、环境、约束放在同一个场景里推演方案。一个完整场景至少包含:

| 要素 | 示例 |
|---|---|
| 用户 | 30 岁出差女白领 |
| 触发情境 | 深夜在机场候机 |
| 目标 | 想在 5 分钟内完成报销提交 |
| 约束 | 网络弱、手机单手操作、低电量 |
| 期望结局 | 不需要打开电脑就能搞定 |

场景分析的价值不在"想象细节",在于:**逼迫设计师承认在现实约束下方案常常不成立**,从而放弃过度理想的方案。

---

## 七、UX 在每个阶段必看的三个判断

无论流程怎么走,资深 UX 永远在反问自己:

1. **为谁解决什么问题?**(没有清晰用户和问题,设计就是自嗨)
2. **依据是什么?**(数据 / 调研 / 理论模型 / 业务目标,不能是"我觉得")
3. **如何验证?**(没有验证手段,等于没设计)
