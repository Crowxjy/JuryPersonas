# PoC 协议 · AI 画像评审设计稿

> 本文档是「AI 画像评审设计稿」能力的概念验证（PoC）协议，**不是产品规范**。
> PoC 的目的是验证一个清晰的假设，通过/不通过都给出明确动作。
> 本协议一旦签字封存，**任何人不得在 PoC 结束前修改判定标准、prompt、IR 提取脚本**。

---

## 一、PoC 假设与判定

### 1.1 项目定位与假设

> **项目定位：AI 画像评审是给设计师提供「更多视角的发现」，而非用研替代品。任何 AI 报告必须经过真人确认后才能进入设计决策。**

> **唯一要验证的假设：AI 画像在「双通道 IR + 视觉」架构下，能在设计师走查的基础上**带来增量有价值的发现**，且误报水平不至于淹没增量信号。**

注意**不是**验证：
- AI 能否替代用研 — 项目定位上就不替代
- AI 报告是否可直接采纳 — 必须经过真人确认
- AI 能否写出"像样的评审" — 这个肯定能

### 1.2 判定标准（封存后不得修改）

| Recall（修正后） | 误报率 FP-hallucination 占 A_set 比 | 结论 |
|---|---|---|
| **≥ 60%** | **≤ 30%** | ✅ 通过，进入 Beta 阶段，可深入投入 |
| 30% ~ 60% | 任意 | ⚠️ 部分通过，仅做"设计师走查辅助工具"，必须人工筛选 |
| **< 30%** | 任意 | ❌ 失败，**至少 6 个月内不再投入"AI 评设计稿"方向** |
| 任意 | **> 50%** | ❌ 失败，AI 编造问题，无论 Recall 多高都不可用 |

**关于发散**：项目允许且鼓励 AI 在 IR 范围之外发散（DV-OOS 类发现），但 AI 输出时**必须显式标识**该条是否可在 IR 中定位。未标识或错误标识的发散一律计入 `FP-hallucination`。

**关于表述**：表述失真（夸大、偷换主语、把主观当客观）一律视为错误，与编造同等计入 `FP-hallucination`。允许发散意味着对表述精度门槛**更高**，不是更低。

**稳健性附加条件**：6 张设计稿的 Recall 标准差 **必须 ≤ 0.15**。否则即便平均达标也判定为"不可预测的不稳定方案"。

### 1.3 实验封存约定

- **D2** 由 **SEC**（实验封存员，由 PM 兼任）将本节判定标准 + IR 提取 prompt + 画像版本号 hash 后封存
- 封存方式：git tag + 加密签名邮件给至少 1 名外部见证人
- **D24 决策会议时由 SEC 当场启封**，对照实测结果

## 二、双通道 IR 架构（评审输入规范）

### 2.1 双通道并行

```
设计稿 ─┬─► 视觉通道 (Vision LLM 看 PNG)         ─► 主观感受/情绪/调性
        │
        └─► 结构通道 (IR 提取，三层)             ─► 客观断言锚点
        
       两条通道 → 同时喂给画像 → 画像评审
```

### 2.2 三层 IR 结构

| 层 | 内容 | 来源 |
|---|---|---|
| **atomic 原子层** | 节点树、bounds、文本、字号、颜色（纯客观） | Figma API 或 vision 抽取 |
| **perceptual 感知层** | visual_hierarchy / reading_path / contrast_warnings / affordance_warnings | 启发式规则 + LLM 推理 |
| **audit 审计层** | uncertainties / potential_blindspots / out_of_scope（**必填**）| LLM 自我审视 |

### 2.3 信息优先级双层

| 层 | 由谁决定 | 价值 |
|---|---|---|
| `intrinsic_priority` | 设计稿本身的视觉权重（大小/颜色/位置/留白/隔离度） | 设计师视角 |
| `contextual_priority` | 由画像在评审前自报"我来这页要做什么" 推导 | 用户视角 |

**洞察规则**：中间模型必须显式标出 `intrinsic_priority` 与 `contextual_priority` 的**错位点**——这才是评审最有价值的产出。

### 2.4 画像评审的硬约束

画像 prompt 拼接时强制注入：

```
1. 主观感受 → 可基于"视觉通道"，但不得超出"第一印象 / 情绪 / 调性"三类
2. 客观断言 → 必须能在"结构通道 IR"找到锚点节点 ID，否则视为编造
3. 遇到 audit.uncertainties 中的项 → 必须明说"看不清，不评论"
4. 允许在 IR 范围外发散（鼓励"更多视角"），但每条发散必须显式打标：
     - [grounded]      可在 IR 中定位
     - [extrapolated]  IR 中不存在，凭经验外推
   未打标 / 错误打标的发散一律视为 FP-hallucination
5. 表述必须精准。夸大、偷换主语、把主观当客观一律视为错误
   （如把"用户心智模型反复重建"作为客观断言而无证据，计 FP-hallucination）
```

## 四、红线清单（违反即作废）

### 4.1 角色互斥红线（绝对不可合并）

| 红线 | 原因 |
|---|---|
| **AI-O ≠ UR** | 操作员看了真人测试就会无意识调 prompt，最常见的自我证明污染 |
| **AI-O ≠ K-\*** | 自己产出的问题自己打指纹/仲裁，等于自己给自己打分 |
| **UR ≠ K-\*** | 听过真人原话再做标注会带主观偏好 |
| **PM ≠ K-\* / AN** | 主理人天然带预设结论，参与任何打分都会渗透 |
| **K-1 ⊥ K-2 同阶段** | 标注/仲裁阶段两人禁止交流（双盲） |
| **EXP ≠ K-\*** | 专家组与仲裁组分开，避免"自家审自家" |
| **AN ≠ AI-O** | 算指标的人若也调脚本，会下意识"美化"数据 |

### 4.2 流程红线

| 红线 | 原因 |
|---|---|
| **prompt / IR 脚本一旦冻结直到 PoC 结束** | 否则就是"在样本上反复调到达标"，不是验证 |
| **EXP 必须先于或独立于 AI-O 完成评审** | 防止专家组看了 AI 报告再"补"评审 |
| **K 跨阶段兼任必须时间隔离 ≥ 1 天** | 减弱"我之前标过这条"的记忆偏差 |
| **判定标准必须封存后再开始实验** | 否则"事后改判定标准"几乎一定会发生 |
| **不许在样本上挑 AI 擅长的场景** | 必须含 AI 不擅长的（如品牌调性），否则结论有偏 |
| **不许扩展样本类型外推结论** | 只对验证过的 3 类页面（列表 / 表单 / 商详）下结论 |

### 4.3 数据红线

| 红线 | 原因 |
|---|---|
| **三个原始集合（A_set/B_set/C_set）必须去标识化后给 K 标注** | K 不能知道哪条来自哪边 |
| **CODER 拿不准的条目必须标 unclassified，不许丢弃** | 防止"难标的就扔了" |
| **AI 报告 / 真人录像 / 专家评审 必须分开存储，访问控制** | 防止跨角色看到"对面的答案" |
| **仲裁日志（K 的 reasoning）必须公开可审** | 防止仲裁本身变成黑箱 |

## 五、问题指纹与重合度计算

### 5.1 问题指纹三元组

每条原始问题压缩为：

```yaml
issue:
  location: btn_buy           # IR 节点 ID（必填）
  category: discoverability   # 15 类枚举（必填）
  severity: high              # high/medium/low（必填）
```

**categories 枚举**（Nielsen 十原则 + 5 项扩展）：
```
visibility / matching / control / consistency / error-prevention /
recognition / efficiency / minimalism / error-recovery / help /
accessibility / hierarchy / discoverability / trust / friction
```

**两条问题视为"同一问题"** ⇔ 同 location（含父子关系）+ 同 category + severity 差距 ≤ 1 档

### 5.2 仲裁标签枚举（K 组产出）

| 标签 | 含义 | 计入 |
|---|---|---|
| `TP` | 双方都报 | 命中 |
| `TP-latent` | AI 报、真人没报，但仲裁确认是真问题 | **命中**（关键修正）|
| `FP-hallucination` | AI 编造 | 误报 |
| `FN-real` | 真人报、AI 真漏 | 漏报 |
| `FN-IR-blindspot` | AI 漏是因为 IR 没采到 | **不归 AI**（归 IR 缺陷）|
| `DIS-AI-correct` | 严重度 AI 更准 | 按 AI 计 |
| `DIS-human-correct` | 严重度真人更准 | 按真人计 |
| `FP-OOS` | AI 报的内容超出当前评测场景，且仲裁认定无价值 | 误报 |
| `DV-OOS` | AI 报的内容超出当前评测场景，但仲裁认定有价值 | **不计入** P/R，单独跟踪 |

### 5.3 主指标公式（修正后）

```
真实命中    = TP + TP-latent
真实漏报    = FN-real
真实误报    = FP-hallucination

Recall      = 真实命中 / (真实命中 + 真实漏报)
Precision   = 真实命中 / (真实命中 + 真实误报)
F1          = 2·P·R / (P+R)
IR 缺陷率   = FN-IR-blindspot / 全部差异条目      # 单独跟踪
```

**严重性加权 Recall**（high=3, medium=2, low=1）：
```
weighted_recall = Σ severity_w(hit) / Σ severity_w(B_set)
```

**类别覆盖率**：
```
category_coverage = |hit_categories| / |B_set_categories|
```

**独家发现价值**：人工评估 TP-latent 中"对真问题贡献"的条目数。

### 5.4 圆桌协同增益

```
synergy_gain = roundtable_recall - max(single_persona_recall_i)
```

如果增益 < 5%，说明圆桌只是冗余，可省。

## 六、样本与对照组

### 6.1 设计稿样本

- **数量**：6 张（不可减），列表页 / 表单页 / 商详页 各 2 张
- **来源**：必须是**已上线、有真实用户反馈数据**的页面（客服记录 / 埋点漏斗 / A/B test 历史）
- **任务脚本**：每张稿配套 1 条核心任务（如"在商详页找到价格并加入购物车"）

### 6.2 三组对照

| 组 | 输入 | 输出 | 角色 |
|---|---|---|---|
| **A 组** AI 评审 | 设计稿 + IR + 画像 prompt | 18 份（6 稿 × 3 次取多数） | AI-O |
| **B 组** 真人测试 | 设计稿 + 任务脚本 + 出声思考 | 30 份（6 稿 × 5 人） | UR |
| **C 组** 专家评审 | 设计稿 + 启发式原则 | 12 份（6 稿 × 2 专家） | EXP-1 / EXP-2 |

### 6.3 控制变量

- 真人看的是和 AI **同一份**渲染图（不是真实 App 体验）
- 真人和 AI **任务目标完全一致**（不要 AI 评首屏、真人评全流程）
- A 组每张稿跑 3 次取多数，去除随机性

## 十、致诚实

本 PoC 的设计原则是**接受失败可能**。如果团队心里已经有"无论结果都要继续做"的预设，就请不要启动这个 PoC——它只会变成表演。

> 如果 Recall < 30% 或 Std > 0.15，立即停止"AI 评设计稿"方向至少 6 个月。
> 这条写不下去，就说明这不是 PoC，是产品设定的合理化包装。
